## ⏰ 详细执行日程与硬核闭环 (Execution Plan)

### 🧱 时间块 1：算法引擎 - 物理时空的“龟兔赛跑” (1.5 小时)
* **【学什么】**：LeetCode 141. Linked List Cycle (环形链表)。判断一个链表中是否隐藏着死循环（环）。
* **【怎么学/去哪练】**：
    1. **物理降维**：绝对不能用 `list` 去记录走过的节点（会导致 $O(N)$ 空间复杂度，且查询极慢）。
    2. **快慢指针架构**：想象在操场跑步。乌龟（`slow`）每次走 1 步，兔子（`fast`）每次走 2 步。
    3. **破局定理**：如果跑道是直的（无环），兔子一定会先到达终点（`fast.next is None`）。如果跑道有环，兔子一定会从后面**“套圈”**追上乌龟（`slow == fast`）。
* **【达标效果与资产沉淀】**：写出空间复杂度为 $O(1)$ 的环形侦测代码。
    * 文件命名：`First_month/day26_linked_list_cycle.py`

### 🧱 时间块 2：Hive 架构 - 降维打击之“炸裂函数” (1.5 小时)
* **【学什么】**：如何把一行数据里的数组（如 `["apple", "banana"]`），物理“炸”成两行独立的数据。
* **【怎么学/去哪练】**：
    1. **业务痛点**：前端传来的埋点日志经常是一个字段里塞了个数组：`user_id: 1, items: [A, B, C]`。如果数仓不把它拆开，根本没法做 `GROUP BY` 统计商品销量。
    2. **语法骨架**：学习 `LATERAL VIEW explode(数组字段) 虚拟表名 AS 虚拟列名` 的魔法。
    3. **本地推演**：在本地测试如何将一行 `(User_1, [Tag1, Tag2])` 转变为两行 `(User_1, Tag1)` 和 `(User_1, Tag2)`。
* **【达标效果与资产沉淀】**：理解“行转列 / 列转行”的数据重塑逻辑。
    * 文件命名：`First_month/day26_hive_explode.sql`

### 🧱 时间块 3：大数据生态 - 异构数据同步底座 DataX 初探 (2 小时)
* **【学什么】**：系统里的业务数据在 MySQL，离线分析要用 Hive，怎么把海量数据每天凌晨搬过去？
* **【怎么学/去哪练】**：
    1. **概念对齐**：了解阿里开源的 **DataX** 或 Apache **Sqoop**。
    2. **底层架构**：理解大厂数据同步工具的核心理念：**Reader（抽水机） -> Channel（水管/内存队列） -> Writer（注水机）**。
    3. **配置逻辑**：去 GitHub 搜一下 DataX 的官方文档，看一眼它的 JSON 配置文件长什么样（不需要立刻安装，重点看懂 JSON 里的 `job.content.reader` 和 `job.content.writer` 是怎么映射的）。
* **【达标效果与资产沉淀】**：在脑海中建立“数据从 RDBMS 流向 HDFS/Hive”的物理管道图。
