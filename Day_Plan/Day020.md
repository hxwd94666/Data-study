## ⏰ 详细执行日程与硬核闭环 (Execution Plan)

### 🧱 时间块 1：Python 架构 - 工业级 DQ 巡检类封装 (3 小时)
* **【学什么】**：面向对象封装 (OOP)、`logging` 标准输出配置、异常捕获后的 I/O 资源回收。
* **【怎么学/去哪练】**：
    1. **类定义**：在 PyCharm 编写 `class DataQualityInspector:`。
    2. **初始化**：在 `__init__` 中利用 `logging.basicConfig()` 强制规定日志输出格式（需包含 `%(asctime)s`, `%(levelname)s`, `%(message)s`）。
    3. **逻辑植入与熔断**：编写 `compare_row_counts(self, source_cnt: int, target_cnt: int) -> float` 方法。人为构造分母为 0 的极端场景，利用 `try-except ZeroDivisionError` 捕获，并通过 `logging.error(msg, exc_info=True)` 将错误堆栈落盘。
    4. **资源释放**：在 `finally` 块中输出 INFO 级别日志，模拟“数据库探针已安全断开”。
* **【达标效果与资产沉淀】**：运行实例后，控制台拒绝一切原生的 `print()`，必须输出带时间戳与级别的专业日志流。
    * 文件命名：`First_month/day20_dq_inspector.py`

### 🧱 时间块 2：算法引擎 - 动态区间的物理降维 (2.5 小时)
* **【学什么】**：LeetCode 3. 无重复字符的最长子串 (Longest Substring Without Repeating Characters)。
* **【怎么学/去哪练】**：
    1. **物理模型**：维护一个由 `left` 和 `right` 指针框定的“动态窗口”。
    2. **哈希防御**：利用 Python 已经学过的 `set` 作为哈希表，记录窗口内是否存在重复字符。
    3. **状态机流转**：`right` 向右扩张吃入字符；一旦发现重复，`left` 开始向右收缩吐出字符，直到报警解除。
* **【达标效果与资产沉淀】**：告别 $O(N^2)$ 的多层循环，实现严格的 $O(N)$ 时间复杂度并在 LeetCode 绿墙通过。
    * 文件命名：`First_month/day20_sliding_window.py`

### 🧱 时间块 3：SQL 布局实例 - 累计与滑动开窗 (1.5 小时)
* **【学什么】**：窗口函数中的 `ROWS BETWEEN` 子句物理意义。
* **【怎么学/去哪练】**：
    1. **场景模拟**：在本地 SQLite/Hive 造数脚本中，模拟某平台用户的每日充值流水表。
    2. **底层查询**：编写 DQL，分别计算“截至当天的历史累计充值（UNBOUNDED PRECEDING）”与“包含今天在内的近 3 天滑动平均充值（2 PRECEDING AND CURRENT ROW）”。
* **【达标效果与资产沉淀】**：代码关键字全大写，瀑布流缩进，精准控制滑动窗口的数据帧（Frame）边界。
    * 文件命名：`First_month/day20_sliding_agg.sql`

## 🗣️ 面试预演 (Interview Simulation)
* **Q**：“在你编写的自动化数据巡检脚本中，如果因为网络抖动导致上游库断连报错，脚本直接崩溃了，你怎么保证占用的下游数据库连接池被正确释放？”
* **A**：“绝对不能依赖 Python 的自动垃圾回收（GC）。我的 DQ 底座是基于 `try-except-finally` 架构的。我会把可能超时的 I/O 操作放在 `try` 中，用 `except` 捕获 Timeout 异常并利用 `logging` 落盘留存案发现场；而释放连接池的 `.close()` 动作，必须硬编码在 `finally` 块中。这样无论是否发生异常甚至主动 return，都能保证连接池被绝对释放，防止 Connection Leak（连接泄露）。”
