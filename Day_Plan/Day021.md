## ⏰ 详细执行日程与硬核闭环 (Execution Plan)

### 🧱 时间块 1：Python 架构 - 企业级日志物理落盘 (2 小时)
* **【学什么】**：`logging.FileHandler` 的多路复用与日志分离输出（Log Routing）。
* **【怎么学/去哪练】**：
    1. **架构改造**：打开 Day 20 的 `DataQualityInspector` 类，在其 `__init__` 初始化方法中引入新组件。
    2. **多管线布局**：创建两根管道。管道 A (`StreamHandler`) 负责控制台全量输出；管道 B (`FileHandler`) 指定输出到本地文件 `dq_error.log`，并强行设置 `setLevel(logging.ERROR)`。
    3. **混沌压测**：运行实例，传入正常数据，检查 `dq_error.log` 是否为空；传入触发除零异常的脏数据，检查红色的报错堆栈是否成功刻录进本地硬盘。
* **【达标效果与资产沉淀】**：控制台能看到所有级别的日志，但本地 `.log` 文件中只沉淀高价值的 ERROR 堆栈快照。
    * 文件命名：`First_month/day21_dq_logger_pro.py`

### 🧱 时间块 2：算法引擎 - 动态窗口的物理缩容 (2 小时)
* **【学什么】**：LeetCode 209. Minimum Size Subarray Sum (长度最小的子数组)。
* **【怎么学/去哪练】**：
    1. **物理模型**：依然是 `left` 和 `right` 构成的滑动窗口，但这次条件从“无重复”变成了“区间 Sum 达到目标值”。
    2. **状态流转**：`right` 向右推土，将元素吃进 ALU 并累加；一旦 Sum $\ge$ Target，触发 `while` 循环，记录当前最短帧长度，随后 `left` 开始向右吐出元素，直到 Sum 再次跌破红线。
* **【达标效果与资产沉淀】**：利用 $O(N)$ 的时间复杂度击穿数组，坚决不使用双层 `for` 循环的暴力拆解。
    * 文件命名：`First_month/day21_sliding_min_window.py`

### 🧱 时间块 3：SQL 架构 - 状态机与连续性步长推演 (2.5 小时)
* **【学什么】**：利用 `DATE_SUB()` 与 `ROW_NUMBER()` 的相对步长特性，计算“用户连续登录/活跃天数”。
* **【怎么学/去哪练】**：
    1. **沙盒造数**：在 SQLite/Hive 测试库中建立一张包含 `user_id`, `login_date` 的登录流水表，人为制造出连续登录 3 天、中断、再连续登录 2 天的数据特征。
    2. **降维推演 (等差数列法则)**：
        * 先用 CTE 对每个用户的 `login_date` 按时间排序打上 `ROW_NUMBER` (如 1, 2, 3...)。
        * **核心核武器**：如果日期是连续的（等差为 1），行号也是连续的（等差为 1），那么 `日期 - 行号` 算出来的“锚点参考日”绝对是同一天！
    3. **聚合收网**：对上述算出的“锚点参考日”进行 `GROUP BY`，计算 `COUNT(*)`，即可得出连续登录的天数。
* **【达标效果与资产沉淀】**：代码关键字全大写，通过多层 CTE (`WITH AS`) 将推演逻辑拆解为：去重 -> 排序打号 -> 计算锚点 -> 聚合筛选。
    * 文件命名：`First_month/day21_consecutive_login.sql`

## 🗣️ 面试预演 (Interview Simulation)
* **Q**：“在生产环境中，你写的自动化清洗脚本为什么要将 INFO 和 ERROR 日志分开落盘，而不是全部写进一个文件？”
* **A**：“这是为了降低底层磁盘 I/O 压力和提升排障的信噪比（SNR）。在海量数据清洗时，INFO 日志的数据量极大，将其全量落盘会快速撑爆服务器磁盘空间，导致服务宕机。因此我采用了日志分流架构：INFO 仅在调试或标准输出中短效保留，而利用 `FileHandler` 单独建立 ERROR 拦截管线。只有致命异常的堆栈快照才会被持久化到独立的 `.log` 物理文件中。这样不仅节省了存储成本，在夜间跑批失败时，报警系统也能第一时间精准抓取独立日志发往钉钉群，实现秒级定位。”
