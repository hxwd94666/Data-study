### 1. 🐍 Python 工程化与算法底座 (Python Engineering & Algo)

#### 1.1 防 OOM (内存溢出) 流式清洗管线
* **物理痛点**：传统 `readlines()` 全量加载会导致处理 GB 级别大文件时服务器内存瞬间撑爆。
* **架构重构**：引入 `Generator (生成器)` 与 `yield` 底层机制。
    * 
    * 摒弃“卡车拉货”模式，建立“流水线传送带”：`for line in file:` 配合 `yield` 逐行挂起并抛出数据。
    * **绝对优势**：无论文件是 10MB 还是 10TB，内存占用永远保持在 $O(1)$（单行数据的体积），实现了天然的 Backpressure（背压）防御。

#### 1.2 高可用 I/O 与脏数据隔离 (Data Quality Gatekeeper)
* **监控流与数据流物理隔离**：
    * 健康数据：通过 `yield` 抛给下游数据库。
    * 脏数据（缺失、类型错误）：触发 `try-except` 熔断，通过 `.strip()` 物理卸妆后进行 `not all()` 校验。
    * **日志多路复用**：利用 `logging` 模块（StreamHandler + FileHandler），拦截致死脏数据并静默写入 `dq_error.log`（保留错误行号与快照），保证主进程绝不宕机。

#### 1.3 算法引擎进阶：双指针降维与链表物理结构
* **多体对撞 (LeetCode 15. 三数之和)**：
    * 掌握“先排序铺设轨道” + “固定单端（降维）” + “双向逼近”的 $O(N^2)$ 架构。
    * **游标滑步去重魔法**：利用 `while nums[left] == nums[left + 1]` 实现指针瞬间跨越重复废料，摒弃了消耗内存的 `set` 去重法。
* **链表架构 (LeetCode 206, 203)**：
    * 
    * 深刻理解链表的“内存非连续存储”特征。掌握 `prev`, `curr`, `next` 三指针在物理内存中的交替指向逻辑，以及利用 `Dummy Node (虚拟头节点)` 解决边界操作痛点。

### 2. 🗄️ SQL 数仓核心架构 (SQL Advanced Architecture)

#### 2.1 会话切分状态机 (Sessionization)
本周攻克了仅次于“连续登录”的第二大复杂业务 SQL 模型：**断点切割**。
* **业务场景**：将用户一天内杂乱的点击流水，按“时间间隔 > 30 分钟”物理切割为多个独立会话。
* **三层 CTE 架构推演**：
    1. **抓取探针**：利用 `LAG(click_time) OVER(...)` 获取上一次行为时间。
    2. **熔断判定**：计算时间差，配合 `CASE WHEN`，当间隔 > 30 分钟或为首条记录时，打上 `1` 的标签，否则为 `0`。
    3. **降维打击**：利用神仙操作 `SUM(标记位) OVER(PARTITION BY user_id ORDER BY click_time)` 进行滚动累加，完美生成绝对唯一的 Session ID。

### 3. 🐘 大数据工具底层逻辑 (Hive Storage Optimization)

针对海量数据的查询缓慢与 JOIN 倾斜问题，完成了 Hive 存储层的物理选型：
* **分区 (Partitioning) - 解决“找得慢”**：
    * 物理层面上，在 HDFS 将数据切分为不同的目录层级（如按天 `dt=20260404`）。
    * 核心优势：利用**“分区裁剪”**，避免 Full Table Scan（全表扫描），极大地减少 I/O 开销。
* **分桶 (Bucketing) - 解决“分布不均”**：
    * 
    * 底层机制：基于特定列（如 `user_id`）的 Hash 值取模，将数据强行打散到多个物理文件（Bucket）中。
    * 核心优势：优化数据抽样（Sampling），并在大表 JOIN 时实现 Map-Side Join，是防御数据倾斜的核心武器。
