## 🏆 核心资产沉淀 (Core Assets)

### 1. 🔥 核武级 SQL：窗口函数防御体系
* **`ROW_NUMBER()` 的排序坍塌防御**：
  * **物理真相**：当 `ORDER BY` 字段存在重复值时，如果不加二级排序字段（Tie-breaker），底层分布式引擎每次跑出的序号会发生随机跳跃（薛定谔的排序）。
  * **规范约束**：强制写出 `ORDER BY score DESC, user_id ASC`，确保结果的绝对幂等性。
* **`LEAD()` / `LAG()` 的偏移量越界处理**：
  * **应用场景**：计算页面停留时长（下一次行为时间 - 本次行为时间）、最大连续活跃天数。
  * **边界防御**：必须使用 `PARTITION BY` 锁定用户作用域，并通过 `LEAD(col, 1, 'default')` 或外层嵌套 `COALESCE()` 函数，防止最后一条记录拉取到 NULL 值引发下游计算的 NPE（空指针异常）。

### 2. 🐍 Python 引擎：OOP 启蒙与 O(1) 极限压测
* **栈结构 (Stack) 的物理降维**：
  * 摒弃了花哨的第三方库，直接利用 Python 底层 `list` 实现栈的 LIFO（后进先出）特性。
  * 深刻理解 `list.append()`（入栈）与 `list.pop()`（出栈）在内存连续分配下的 $O(1)$ 极速性能。
* **Hash 映射对抗面条代码**：
  * 在处理 LeetCode 20（有效的括号）时，利用 `dict` 建立 `mapping = {')': '(', '}': '{', ']': '['}`。
  * 成功将冗长脆弱的 `if-elif-else` 硬编码，重构为高扩展性的 $O(1)$ 字典查询。

### 3. 🧠 DE 底层视野：分布式网络风暴
* **Hash 数据倾斜 (Data Skew)**：理解了当海量相同 Key（如海量 NULL 值或大促热点商品）经过 Hash 取模后，会全部涌向单一 Reducer 节点，导致单机 OOM（内存溢出）。
* **Shuffle 过程的昂贵代价**：明确了 Map 端到 Reduce 端的数据重分布是集群计算中最消耗 I/O 和网络带宽的阶段，建立起“尽可能在 Map 端提前聚合”的 SQL 优化嗅觉。

## 🗣️ 面试高频话术 (Interview Defense)
* **Q**：“怎么计算某个用户的最大连续活跃天数？只准用一条 SQL。”
* **A**：“核心思路是利用等差数列特性。首先用 `ROW_NUMBER()` 按日期排序打上连续递增的行号；然后用真实的 `登录日期` 减去这个 `行号` 得到一个『基准日期』。如果用户是连续登录的，他们减出来的基准日期必然是同一天。最后对这个基准日期进行 `GROUP BY` 并 `COUNT()`，求出最大的 count 值即可。同时，为了防止同一个用户一天内多次登录干扰逻辑，我在最内层子查询会先做一次 `DATE(login_time)` 的去重处理。”
